## What E03b Found

The navigator hypothesized fusion wins at FinTek sizes because Python overhead between ops would dominate. Right about the direction, wrong about the mechanism.

**Actual mechanism**: CuPy's dispatch cost per call (10-27us each). Not Python overhead between calls (0.1us — negligible). Reducing 17 calls to 8 saves ~180us on a ~280us total. That's the 2x.

| Size | A: Independent | C: Fused | Speedup |
|------|----------------|----------|---------|
| 50K  | 0.280ms | 0.135ms | 2.08x |
| 100K | 0.288ms | 0.143ms | 2.01x |
| 500K | 0.298ms | 0.198ms | 1.50x |
| 900K | 0.358ms | 0.262ms | 1.37x |
| 10M  | 1.027ms | 2.263ms | 0.45x (regression) |

Shared intermediates (Path B) win at ALL sizes — 1.2-1.5x consistently. No crossover. The safest always-win optimization.

**The 10M regression is a kernel quality problem, not a conceptual failure.** The fused kernel achieves ~200 GB/s vs CuPy's 1,200 GB/s at large sizes. Proper occupancy tuning (grid-stride loop, adaptive block count) would close this gap.

**Rust shifts the crossover dramatically.** CuPy dispatch = 70us/call. Rust dispatch = 9us/call. At 9us, the dispatch savings from fusion shrink (9 calls saved x 9us = 81us saved), but the kernel bandwidth advantage takes over. Crossover moves from ~500K toward 5M+.

## The Corrected Compiler Architecture

Replacing Entry 009's premature "one trap" conclusion with the measured one:

1. **Shared intermediates — always.** Share cumsums, sorts, any primitive that appears in multiple leaves. 1.2-1.5x at all sizes. CSE over the primitive DAG achieves this.

2. **Element-wise fused kernels — always.** Any expression tree that maps elements independently fuses into one kernel. 2.3x from E05. No crossover.

3. **Cross-operation fusion — size-adaptive.** Below ~500K rows: fuse (2x from dispatch reduction). Above ~1M rows: compose CuPy primitives (bandwidth wins). The compiler needs a size-based cost model.

4. **Rust dispatch layer shifts everything.** At 9us/launch, cross-operation fusion crossover moves much higher. The fusion benefit shrinks but composition cost also shrinks.

## What the Registry Must Express (E04 Spec)

Four properties per specialist:

```
specialist: rolling_z_score
inputs: [data, window]
primitive_dag:
  - cumsum(data)    -> cs      # named output — shared with other specialists
  - cumsum(data^2)  -> cs2     # named output — shared with other specialists
  - window_mean(cs, window)    -> mean
  - window_var(cs2, cs, window) -> var
  - sqrt(var)       -> std
  - (data[w-1:] - mean) / std -> z_out

fusion_eligible: true           # can be fused with adjacent specialists
fusion_crossover_rows: 500_000  # above this, prefer composed primitives
independent: true               # reorderable (no scan ordering constraint)
```

The compiler uses this to:
1. Build merged DAG, run CSE, find shared primitives
2. For fusion-eligible subgraphs below crossover: generate fused kernel via NVRTC from Rust
3. For subgraphs above crossover: compose CuPy primitives, share intermediates only

This is not premature complexity — E03b proved it necessary. Leave it out and you have a compiler that's 0.45x at large sizes OR misses 2x at small sizes.

## E04 First Implementation Target

Simplest prototype that proves the full loop:

**Input**: Pipeline spec with two specialists sharing a cumsum
**Output**: Single fused CUDA kernel, NVRTC-compiled from Rust, correct output

**What this proves**: spec -> primitive DAG -> CSE -> code generation -> NVRTC -> launch -> verify

Even for one specialist, this is the full loop. Add a second that shares the cumsum — CSE fires and eliminates the redundant computation. That's the demonstration.
