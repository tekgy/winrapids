## E09: What We Now Know (Confirmed)

Rust NVRTC is faster than expected in both dimensions that matter:

**Compilation**: 22ms (Rust) vs 40-92ms (CuPy) — 2-4x faster. The gap is Python overhead: dict lookups, source hashing, option string construction, C extension crossing. Rust eliminates all of it.

**Launch overhead**: 9μs (Rust) vs 70μs (CuPy) — 7.7x faster. At 100K leaves/cycle this is 6.1 seconds of pure Python tax eliminated. For empty kernels, both converge to ~7-8μs — the raw WDDM hardware floor.

**Full pipeline verified**: source string → NVRTC → PTX → cuModuleLoadData → launch → verify. Zero issues on Windows + CUDA 13.1 + WDDM + Blackwell sm_120.

### What E09 Changes for E03b Interpretation

The 7.7x launch advantage reshapes the fusion calculus. At 100K rows, 17 CuPy calls → 8 fused calls = 9 fewer launches × 70μs = 630μs saved from dispatch alone, before any bandwidth benefit. That's on top of compute that takes ~0.07ms total. Fusion could win purely on dispatch reduction even if the kernel itself isn't faster.

In Rust, that advantage shrinks (9μs × 9 = 81μs saved) — but the compute is faster too, so the relative picture changes again. This is why E03b needs to be measured, not concluded.

## The E03b Gap

**Lab notebook Entry 009 draws this conclusion:**
> "Cross-operation fusion: usually LOSES. Custom kernels are 2.4x SLOWER than CuPy primitives."
> "The pipeline generator should NOT be a cross-operation kernel fuser."

**This conclusion was written before E03b ran. E03b has not been executed.**

Code is at `experiments/E03-cross-algorithm-sharing/bench_e03b_realistic.py`. Ready to run.

The specific problem: E03 ran at 10M rows in a tight CuPy loop. At 10M rows:
- Each CuPy call takes ~0.2ms — Python overhead (70μs) is ~35% of execution time
- Blackwell's 96MB L2 keeps all data resident — bandwidth savings disappear

At FinTek's actual sizes (50K-900K rows), each CuPy call takes ~0.07ms — Python overhead EXCEEDS compute time. 17 CuPy calls × 70μs = 1.19ms overhead on 0.07ms compute. Fusion → 8 calls × 70μs = 560ms overhead. That's 630μs improvement from dispatch alone. This is ~9x the compute cost itself.

Entry 009's "one trap" section should carry the annotation: **PENDING E03b at FinTek sizes.**
