## Proposals

### 1. TamSession producer wiring — Tier 1 (architecture)

We added 13 IntermediateTag variants this session. All are declared but none have producers
except DelayEmbedding (wired via the session-aware complexity functions). The Tier 1 tags
are high-value NOW because every fintek bin run is wasting them:

- **SortedArray**: shapiro_wilk and quantile both sort independently. Wire shapiro_wilk to
  register, quantile to check-before-sort.
- **FFTOutput**: every Family 6 leaf runs FFT independently on the same bin. Wire periodogram
  to register, spectral entropy / spectral centroid / etc. to check-before-FFT.
- **CovarianceMatrix**: pca, factor_analysis, manova each compute independently. Wire pca to
  register, others to check-before-compute.

**What I'd seed**: `tamsession-tier1-producers` campsite under architecture. This is the
concrete implementation work that turns the phyla map from documentation into speedup.

The pattern is established (complexity.rs session variants are the template). This is
mechanical wiring with a clear before/after: run `cargo bench` on a full fintek bin,
count FFT invocations before and after.

### 2. Kingdom decidability — does the recurrence definition determine the Kingdom? (theory)

The PELT session surfaced this: the tropical semiring recurrence is Kingdom A, but PELT's
pruning optimization makes the full algorithm Kingdom B. This is a general phenomenon:

> *An adaptive algorithm that prunes or updates based on intermediate DP values is Kingdom B
> even when its underlying recurrence is Kingdom A.*

The question: is there an algorithm that takes a recurrence definition (expressed in some
formal language) and outputs its Kingdom? If yes, the spec compiler could issue Kingdom
certificates. If no (undecidable in general), the question becomes: what's the decidable
fragment?

What I know:
- The affine semigroup (EMA, GARCH filter) is decidable Kingdom A: check if operators compose
  as (a,b)∘(c,d) = (ac, ad+b) with bounded representation.
- The tropical semiring is decidable Kingdom A for the *recurrence*. The algorithm may be B.
- Wynn's epsilon is Kingdom BC: anti-diagonal tableau structure, non-liftable.

**What I'd seed**: `kingdom-decidability-question` under theory — owned by Aristotle or
math-researcher. The product is a formal characterization of the decidable fragment, with
proofs that each algebra in the Op enum is correctly classified.

This matters for the spec compiler: if we can decide Kingdom from the recurrence definition,
the compiler can issue certificates without human annotation.

### 3. The singularity-as-identity bug class — systematic audit (rigor)

The adversarial agent named this in wave 18: functions that return `eye()` or some identity
structure for singular/degenerate inputs, producing plausible-looking but wrong answers.
This is harder to catch than NaN propagation because the output looks valid.

Examples I'd expect to find:
- `covariance_matrix` of a constant column: should signal degeneracy, not return a zero matrix
- `eigendecomposition` of a singular matrix: may return identity eigenvectors silently
- `correlation_matrix` with a pair of identical columns: rank-deficient, but output looks full-rank

**What I'd seed**: `singularity-as-identity-audit` under rigor — owned by adversarial.
The test oracle is: for any input that triggers a singularity path, the output must either
(a) return NaN/Err at the degenerate position, or (b) document that identity is the correct
mathematical answer for that degenerate case. No silent plausibility.

### 4. IntermediateTag compatibility enforcement — the sharing contract (architecture)

The phyla map audit surfaced a correctness invariant that isn't enforced in code: two methods
asking for "the covariance matrix" may have different ddof requirements. The tag carries `ddof`
but nothing stops a consumer from accepting a `ddof=0` matrix when it needs `ddof=1`.

The current `TamSession::get` is untyped beyond the Rust type — it checks `Arc<T>` downcast
but not the tag's semantic parameters. A consumer could call `session.get(&wrong_tag)` by
mistake and get the wrong ddof silently.

**What I'd seed**: `tamsession-compatibility-enforcement` under architecture. The product:
a `is_compatible_with` method on each intermediate type (already exists on DistanceMatrix),
and a session `get_compatible` that checks compatibility before returning.

This is Option B from the task description — lightweight, doesn't need the full DAG, just
per-type compatibility checks.

### 5. The explore() epistemic state (theory / TBS)

Beyond `using()` (I know the parameter) and `discover()` (find the best value), there's a
third state: **deliberate refusal to collapse**. Not "I haven't decided" but "I commit to
maintaining superposition until computation reveals the cost of collapsing."

This is different from `discover()` which collapses to a winner. `explore(param)` says:
"run all variants, report all results, never pick." The user sees the full landscape.

The holographic screen connection: maintaining superposition IS the error-correcting
structure. Collapsing early is information loss. `explore()` is the TBS surface for
"give me the holographic screen."

**What I'd seed**: `explore-epistemic-state` under tbs or theory — half-formed. The product
might be a formal spec for what `explore()` returns (a Superposition that never collapses?)
and how it differs from `.discover()` semantically.

## My read on priorities

**Must-happen-next (blocks real speedup)**: TamSession producer wiring (#1). Without
producers, the 13 new tags are documentation, not cache hits. The FFT sharing alone is
worth 15-20x on fintek Family 6 per bin.

**Highest intellectual leverage**: Kingdom decidability (#2). If solved, the spec compiler
gains the ability to auto-classify recurrences. This is a theorem, not just code.

**Rigor debt that compounds**: Singularity-as-identity audit (#3). Unlike NaN (produces
obviously wrong values), singularity-as-identity produces *plausible* wrong values. They
survive into downstream analysis silently. The longer we wait, the more downstream code
relies on the wrong answers.

**When we have bandwidth**: compatibility enforcement (#4) + explore() spec (#5).

## One thing I noticed that nobody named yet

The `superposition.rs::sweep_stationarity` function had 8 compile errors from `.p_value`
references on structs that don't have that field. This means the stationarity superposition
has been broken (silently, due to the linker error masking compile errors) since it was
written. Anyone calling `sweep_stationarity` on a real fintek bin would get a compile failure.

This suggests a systematic risk: functions in superposition.rs may reference struct fields
that were renamed or never existed. An audit of ALL field accesses in superposition.rs
against the actual struct definitions in time_series.rs would be valuable.

**What I'd add to the wave 18 adversarial sweep**: struct-field-access audit in superposition.rs.
