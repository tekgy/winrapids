## What was done

Hand-wrote the three reference programs in the text encoding defined by the spec:
- `programs/sum_all_add.tam` — single-slot sum accumulation
- `programs/variance_pass.tam` — three-slot accumulation (sum, sum_sq, count)
- `programs/pearson_r_pass.tam` — six-slot accumulation for Pearson r

These are the parser/printer test fixtures. The parser round-trip test (1.8)
will parse each of these and assert the printed output equals the original text.

---

## What almost went wrong

**pearson_r: too many phi registers.** The pearson_r kernel has six loop-carried
accumulators, each with a prime-suffix pair. When writing it out I almost
confused `%acc_sx'` with `%acc_s2x'` — the names are similar enough that a
typo would silently produce wrong math (summing x into the sum_x2 slot). The
lesson: when there are many phi pairs, name them descriptively, not numerically.
Descriptive names (`%acc_sx`, `%acc_sxy`) are harder to confuse than `%acc3`,
`%acc5`. The spec should enforce this in the verifier: phi-register base names
must be distinct from each other.

**variance: the `%one` inside the loop.** I put `const.f64 1.0` inside the
loop body (as `%one`) rather than hoisting it to `entry`. This is correct per
the spec (const is a pure op with no side effects, safe to re-evaluate per
iteration), but a backend might naively re-emit a `mov.f64` instruction every
iteration. A future optimization pass should hoist loop-invariant consts. Filed
in `future-ops.md` under "loop-invariant code motion."

## What tempted me off-path

**Adding a `mean_pass` fourth fixture.** Mean is so simple (sum / count, which
is just variance_pass restricted to slots 0 and 2) that writing it would take
30 seconds. But the directive says three programs, and mean doesn't exercise
anything that variance_pass doesn't already cover. Deferred. If a fourth
fixture is wanted, it can be added during 1.7 when the parser needs stress testing.

**Writing `pearson_r_pass` with the final formula inlined.** Real Pearson r
needs a `fsqrt` and two `fdiv` calls, plus a subtraction — that's a gather
operation, not an accumulate operation. The temptation was to make the kernel
self-contained by doing the final formula inside the loop somehow. But the
contract is: kernels accumulate, the host gathers. Sticking to that contract
keeps the IR clean and is exactly the accumulate+gather decomposition that
invariant I7 mandates.

---

## What the next traveler should know

**The slot index constants.** Every kernel ends with a block of `const.i32`
declarations for slot indices, then `reduce_block_add` calls. This pattern
will be the same in every reduction kernel. When you're writing more kernels,
notice the repetition — it's a smell that suggests "slot index" might eventually
warrant a cleaner syntax. For Phase 1, the explicit const is fine.

**The loop-carried register count grows with recipe complexity.** sum_all_add
has 1 phi pair. variance_pass has 3. pearson_r_pass has 6. A full correlation
matrix across many columns would have O(n^2) phi pairs. That's a scale concern
for Phase 2, not Phase 1 — but it's worth knowing that "number of phi pairs"
is a complexity metric for kernels.

**The programs/ directory is the canonical test fixture set.** When the parser
is built, it should read these files directly (not inline strings) so that
editing the `.tam` files updates the tests automatically.
