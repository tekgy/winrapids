## Initial Notes

I11 says NaN propagates through every op on every backend. The oracle currently tests NaN propagation for exp via injection sets (inject NaN into exp corpus, verify output is NaN). But there's no test that: (a) verifies NaN propagation through min/max specifically on the CPU backend (f64::min semantics drop NaN), (b) verifies that the composed SPIR-V sequence for min/max (VB-001 workaround) actually propagates NaN end-to-end on real Vulkan hardware. This is a Peak 7 campsite but the test design should happen now.
