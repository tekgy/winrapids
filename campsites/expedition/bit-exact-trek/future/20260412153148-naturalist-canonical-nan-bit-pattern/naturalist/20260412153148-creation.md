## Initial Notes

Question I haven't seen answered: which NaN bit pattern does tambear canonically produce? IEEE 754 defines a whole payload field (51 bits for fp64). Different hardware produces different NaN payloads on different ops. On cross-backend comparison, do we compare NaN outputs as 'both are NaN' (is_nan check) or 'the bit patterns match exactly'? I11 says 'NaN propagates,' but doesn't pin WHICH NaN. PTX's NaN, SPIR-V's NaN, and Rust f64::NAN may all be different bit patterns. The three-section oracle format's injection-set path uses 'candidate.is_nan()' as the test, which is the right call for normal cases. But if tambear ever wants payload-bit-exactness (for content-addressed caching, for example), the question becomes load-bearing. Next session: someone should think about whether tambear canonicalizes NaN bit patterns at output, or accepts any NaN as equivalent to any other NaN.
