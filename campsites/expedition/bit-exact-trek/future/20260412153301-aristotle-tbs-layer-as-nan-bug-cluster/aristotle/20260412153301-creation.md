# aristotle-tbs-layer-as-nan-bug-cluster

Created: 2026-04-12T15:33:01-05:00
By: aristotle

## Initial Notes

Every adversarial pitfall P16 through P20 from the 2026-04-12 baseline sweep was in the TBS (scripting) layer, NOT in tambear-primitives. P17 Sign(NaN), P18 Min/Max(NaN, x), P20 Eq hardcoded epsilon — all are in crates/tambear-primitives/src/tbs/mod.rs. tambear-primitives' direct math paths were clean. Is there a STRUCTURAL reason NaN bugs cluster in TBS? Hypothesis: TBS is a scripting layer that re-implements math ops in a separate code path from the primitives, and the re-implementation didn't inherit the NaN discipline that the primitive layer had. If true, this is a layering issue: the primitive layer enforces IEEE-754 compliance, TBS bypasses the primitives and re-derives results. Check: does TBS actually bypass tambear-primitives? If yes, TBS should either (a) be rewritten to call through the primitives, or (b) be certified independently to I11. If no, the clustering is coincidence. Small investigation but potentially architectural.
