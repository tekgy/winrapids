## Initial Notes

Fifth deconstruction target I never ran. 'Why SSA for the .tam IR?' — SSA (static single assignment) is the trek's IR form but has never been audited. Could be: (a) a first-principles choice (SSA is the minimum form that makes data dependencies explicit), (b) an inherited convention from LLVM/MLIR dominance, (c) a type-system fact that follows from the two-operation decomposition. Run the full eight-phase cycle. Specifically: is SSA another convention-to-declaration finding (the convention is 'each value is assigned once' and the declaration is 'explicit virtual registers') or does it produce a DIFFERENT move-shape? This is the best single target for testing method bias because SSA is structurally distinct from the previous four targets. If SSA deconstruction also produces a convention-to-declaration move, the method has bias. If it produces something else, the method is finding what's there.
