## Initial Notes

The '50 digits' in mpmath's reference precision for libm validation is a magic number I flagged in I9 Phase 1 assumption autopsy but never deconstructed. Why 50? Not 34 (next IEEE level, quad precision), not 100, not 1000. Probably because it's 'enough for fp64 + margin' — 50 decimal digits ≈ 166 bits, far above fp64's 52 mantissa bits, so rounding to fp64 is unambiguous for most inputs. But for Table Maker's Dilemma inputs (close to fp64 midpoints), 50 is NOT enough — those can require thousands of digits to resolve. Run a mini-deconstruction: what does the choice of 50 optimize? Is there a better choice? What's the per-function minimum precision needed to guarantee correctly-rounded verification? The answer probably varies per libm function and is a campsite input to scientist's oracle runner design.
