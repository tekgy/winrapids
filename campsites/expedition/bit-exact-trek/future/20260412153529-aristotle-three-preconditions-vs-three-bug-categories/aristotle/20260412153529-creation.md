## Initial Notes

Structural observation I noticed but didn't write up: there's an ISOMORPHISM between the meta-goal's three preconditions (IR-precision, faithful-lowering, IEEE-754 compliance) and the three categories of bugs I mentioned in the synthesis.md 'bug triage' usage guide: (a) implementation disagrees with mathematics = IR-precision violation / math bug; (b) backend disagrees with reference = faithful-lowering violation / backend bug; (c) hardware disagrees with IEEE-754 for ops used = precondition-3 violation / hardware bug. Each precondition has exactly one category of bug that violates it. That's not a coincidence — it's two framings of the same three-axis split: the precondition is the claim, the bug is the failure, each precondition has a dedicated failure mode. This could be a useful framing for the bug triage process if navigator wants to formalize a bug-to-precondition mapping. Minor but structurally clean.
