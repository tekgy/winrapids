## Initial Notes

Observation about I3: we forbid FMA contraction because it introduces cross-backend drift. But FMA is ALSO load-bearing POSITIVELY — specifically, FMA is what enables the error-free transforms (TwoSum, TwoProd) used in compensated arithmetic and double-double. An EXPLICIT FMA op in the .tam IR (as opposed to implicit contraction) would be a useful primitive for tambear's eventual double-double work or for any error-compensated algorithm. The distinction: 'fmadd.f64 %dst = %a, %b, %c' as an explicit op that the user/libm author requests vs 'mul.f64 then add.f64' silently getting fused by the backend. If tambear Phase 2 wants correctly-rounded libm, it will probably want an explicit fmadd op, and I3 forbids only the IMPLICIT version. Worth noting that I3 does not close off FMA for the library — it closes off COMPILER-AUTOMATIC contraction. Pathmaker should consider an explicit fmadd.f64 IR op at some point.
