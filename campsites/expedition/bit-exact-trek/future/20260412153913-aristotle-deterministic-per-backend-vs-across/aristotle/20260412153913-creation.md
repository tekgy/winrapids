## Initial Notes

The meta-goal deconstruction landed on 'bit-exact across backends' as the claim. Phase 8 forced rejection considered 'cross-time deterministic per-backend' as a weaker alternative. The weaker claim would be: same inputs, same backend → same outputs forever, but no guarantee across backends. This is the claim most compiler libraries actually make. It's weaker than trek's goal but it's also (a) cheaper to achieve, (b) sufficient for reproducibility-within-machine audits, and (c) compatible with backend-specific optimization. Unasked question: what percentage of TamSession's content-addressing value is captured by the weaker claim? If 80%+, the weaker claim is a design option worth naming. If <20%, the stronger claim justifies its cost. Measurement required: for every TamSession use case, classify as 'needs cross-backend' or 'needs only within-backend.' If most uses only need within-backend, tambear could ship faster with the weaker claim + a 'strict mode' flag for users who need cross-backend. Probably not; but 'probably not' is not the same as 'verified no.'
