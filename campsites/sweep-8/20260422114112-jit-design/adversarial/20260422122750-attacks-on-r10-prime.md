# Adversarial attacks on R10′

**Author:** adversarial · **Date:** 2026-04-22  
**Target:** aristotle's phase-1-8-deconstruction.md

Three assigned attacks + three bonus attacks. All produce either a concrete
breaking scenario or a confirmed survival. No rhetorical attacks — only ones

## ATTACK 1 — NPU graph compilers: `lower(ir, shape)` per-atom contract breaks

**Verdict: R10′ fails. The NPU door cannot fit the per-atom `lower()` contract.**

### The break

R10′'s `DoorCodegen::lower` takes one `TambearIr` (one atom: one accumulate or
gather) and returns one `CompiledArtifact`. The contract is atom-granular.

NPU graph compilers (AWS Inferentia / `neuron-cc`, Google TPU / XLA, Cerebras
Wafer-Scale Engine / CS-2 SDK) reject this contract at the architecture level:

- **Inferentia `neuron-cc`** compiles a WHOLE computation graph (an XLA HLO
  graph or TensorFlow GraphDef) to a single `.neff` binary. It cannot compile
  a single accumulate atom independently — the whole subgraph must be present
  for partition assignment across the 128 NeuronCores. A single `Add + All`
  has no meaningful standalone compilation unit for this architecture.

- **Google TPU / XLA**: XLA takes an `HloModule` (the full computation graph),
  runs a global fusion pass that collapses operations across boundaries, then
  produces a compiled binary. Single-op compilation is not an XLA primitive.
  The entire value proposition of TPU is that the compiler sees the WHOLE graph
  and fuses across arbitrary op boundaries.

- **Cerebras CS-2**: the "layer pipelining" mode assigns different layers to
  different spatial regions of the 850,000-core wafer. It is physically
  impossible to compile one accumulate independently — spatial assignment
  requires knowing the full graph topology.

### Aristotle's draft answer, precisely wrong

> "NPU graph compiler IS the codegen; the whole pipeline becomes one binary;
> dispatch is one call. Survives R10′."

This misses the granularity mismatch. The NPU door's `lower()` receives atoms
one at a time (because that's what `DoorCodegen::lower` accepts). To produce
its whole-graph binary, the NPU door would have to:

1. Buffer incoming atoms without compiling.
2. Wait until it has "all the atoms in this pipeline."
3. Compile them all together.
4. Return the same `CompiledArtifact` for every `lower()` call from this pipeline.

This is not how R10′ is designed. There is no concept of "compile these atoms
together" in the trait. Each `lower()` call is independent. The NPU door has
no way to know when it has received the "last atom of this pipeline" — there is
no pipeline-boundary signal in R10′.

### The fix R10′ needs

A `lower_pipeline` method that accepts a whole `Vec<TambearIr>` (the full
pipeline's atoms) rather than a single atom. The `lower()` per-atom method
becomes a fast path for doors that CAN compile atoms independently (CPU, GPU);
`lower_pipeline()` is the necessary path for graph-compiler doors (NPU).

```rust
trait DoorCodegen {
    // Per-atom compile. Valid for CPU/GPU. NPU default impl: Err(NotSupported).
    fn lower(&self, ir: &TambearIr, shape: &Shape, cap: &DoorCapability)
        -> Result<CompiledArtifact, CompileError>;

    // Whole-pipeline compile. Required for NPU doors.
    // Default impl: calls lower() for each atom independently.
    fn lower_pipeline(&self, atoms: &[TambearIr], shapes: &[Shape],
                      cap: &DoorCapability)
        -> Result<Vec<CompiledArtifact>, CompileError> {
        atoms.iter().zip(shapes).map(|(ir, s)| self.lower(ir, s, cap)).collect()
    }
}
```

NPU doors override `lower_pipeline()` and return `Err` from `lower()`. CPU/GPU
doors use the default `lower_pipeline()` which delegates per-atom. The pipeline
compiler (Sweep 23) calls `lower_pipeline()` for NPU doors and `lower()` per
atom for GPU/CPU doors.

**Implication for DEC-019**: the "per-recipe kernel" model holds for CPU/GPU.
NPU doors compile per-pipeline. Both are first-class DEC-019 patterns. The
single-binary-per-pipeline behavior aristotle describes IS achievable, but
requires `lower_pipeline()` as the seam — not a workaround inside a per-atom
`lower()` call.

## ATTACK 2 — DeterminismClass needs a 4th bucket: SeededDeterministic

**Verdict: R10′ fails. The three-class enum has a gap for seeded-RNG dispatch.**

### The break

R10′ has `DeterminismClass { Deterministic, OrderDependent, NonDeterministic }`.

Consider `Probabilistic` grouping where the weights come from a temperature-
scaled softmax and the user supplies an explicit seed:

```
attention_pool(col=0, weights=softmax(col=1), seed=42)
```

with `using(seed=42)`. The seed bakes into the param hash and the kernel is
compiled with the seed embedded in the IR.

- Run 1 with seed=42: outputs `[0.312, 0.561, ...]`. Deterministic given seed=42.
- Run 1 with seed=99: outputs `[0.841, 0.203, ...]`. Different but deterministic.
- Run 2 with seed=42: identical to Run 1. Reproducible. ✓

This is NOT `NonDeterministic` (it IS reproducible). It is NOT `Deterministic`
in R10′'s sense (unconditionally same output for same input — the seed changes
output without changing input data). It is NOT `OrderDependent` (seed does not
depend on element order).

### The actual 4th class

`SeededDeterministic(seed_hash: u64)`: reproducible given the same seed, but
the seed is the controlling variable. The seed participates in the param hash
(so it bakes into the cache key), but `DeterminismClass` must carry it so
downstream consumers can reason about reproducibility:

```rust
pub enum DeterminismClass {
    /// Same input data + same door + same shape → identical output on every run.
    Deterministic,
    /// Same input data in same element order → identical output; reordering
    /// may change result (tiebreak-sensitive ops like ArgMax).
    OrderDependent,
    /// Output may differ across runs for the same input (GPU warp scheduling,
    /// CUDA atomics without deterministic mode).
    NonDeterministic,
    /// Identical output given the same RNG seed; different seeds produce
    /// different outputs. Seed is baked at compile time via `using(seed=N)`.
    SeededDeterministic { seed_hash: u64 },
}
```

### Second 4th-class candidate: ConvergenceDependent (EM / MCMC)

EM/MCMC algorithms: output is reproducible given the same seed AND the same
convergence path. But convergence order is platform-dependent. This is strictly
weaker than `SeededDeterministic` — it degrades to `NonDeterministic` unless
the convergence path is fixed by the seed AND a deterministic convergence order
is enforced.

The correct answer: EM declares `NonDeterministic` (conservative). User can
force `SeededDeterministic` via `using(seed=N, deterministic=true)` at the cost
of requiring serial convergence for bit-exactness. The `DeterminismClass` in
the compiled artifact reflects the actual determinism guarantee, not the intent.

## ATTACK 3 — Validity::Error with async dispatch: error masking

**Verdict: R10′ has a gap. Multiple in-flight dispatches can mask validity errors.**

### The stated contract

Aristotle: "`wait()` returns `Err(LaunchError::Validity(details))`."

This is correct for the single-dispatch case. The break appears in the
multi-dispatch (pipelined) case.

### The break: error masking under pipelined dispatch

```
stream = backend.default_stream()
e1 = dispatch(stream, kernel_A, inputs_with_NaN, ...)    // NaN → Validity::Error
e2 = dispatch(stream, kernel_B, output_of_e1, ...)        // enqueued immediately

// Host continues without waiting (TAM's natural pipeline scheduling)
let r1 = wait(e1)   // Err(LaunchError::Validity(...))
let r2 = wait(e2)   // Ok(()) — kernel_B ran successfully
```

**What actually happens on CUDA (the reference GPU door)**:

`cuLaunchKernel` returns immediately. A Validity::Error is a numerical event
produced INSIDE the kernel — the kernel runs to completion, writing a sentinel
or NaN into output buffers, then signals its event. `cuLaunchKernel` for
kernel_B is called while kernel_A is still running and kernel_B is enqueued on
the stream. kernel_B WILL execute after kernel_A finishes — CUDA streams do
not abort on kernel "numerical errors" (they only abort on launch errors like
OOM).

**The result**: kernel_B executes with NaN-poisoned data from kernel_A.
kernel_B's output is semantically garbage. But `wait(e2)` returns `Ok(())`
— kernel_B ran successfully from the GPU's perspective. The Validity::Error
from kernel_A is returned by `wait(e1)`, but the caller has already enqueued
and "succeeded" on kernel_B.

For TAM's natural pipeline scheduling (submit multiple steps before checking
any result), this means: step 36's "successful" output is garbage because step
35 hit a validity error. The user gets `Ok(())` for step 36 and has no reason
to distrust it.

### Bonus question: cache entry corruption

**Is the cached kernel corrupted by a Validity::Error?**

No. The binary is correct — it correctly detected the NaN and produced the
sentinel output. `DoorCache::get(key)` returns the same `Arc<Loaded>` for
subsequent calls and it remains valid for non-NaN inputs. Cache is not at risk.
This part of the worry can be closed.

### What R10′ needs: stream-poison semantics

After `wait(e)` returns `Err(Validity)`, the stream must be marked poisoned.
Any subsequent `dispatch()` on the same stream returns
`Err(LaunchError::StreamPoisoned)` without issuing the dispatch to the driver.
Add:

```rust
trait DoorDispatcher {
    // ... existing methods ...

    /// Reset a poisoned stream. Drains any queued work and returns the stream
    /// to a clean state. Must be called after handling a Validity error before
    /// the stream can accept new dispatches.
    fn stream_reset(&self, stream: &Self::Stream) -> Result<(), LaunchError>;
}
```

The stream-poison contract closes the gap: downstream dispatches on the same
stream are blocked until the caller explicitly resets. This forces the caller
to handle the Validity error before proceeding.

**A weaker alternative**: require TAM to call `wait(e1)` before `dispatch(e2)`
when `kernel_A` has `Validity::Error` risk. This moves the responsibility to
TAM (Sweep 23's pipeline scheduler) rather than the trait. The trait documents
the gap; TAM's scheduler fills it.

## BONUS ATTACK 4 — Validity policy vs DenormalMode: the intersection

**Verdict: Gap exists. DenormalMode is not in the cache key; compiled artifacts can mismatch.**

### The break

User writes `using(validity=Propagate)` expecting exact NaN handling. The
door's `DenormalMode` is `FlushToZero` (GPU default). Under FTZ:

`1e-310_f64 + 0.0` → `0.0` (denormal silently zeroed before the lift step
even runs).

`Validity::Propagate` is about NaN/Inf — it says nothing about denormals.
`1e-310` is finite, so `NoNonFinite` AssumptionTag could still be applied.
The kernel is compiled assuming finite inputs, FTZ is active, and denormals
become 0 silently. The user observes wrong results with no error or warning.

**Specifically**: `Welford` running variance on a dataset containing denormal
values produces a wrong mean and variance under FTZ, even with
`Validity::Propagate`. The denormals are zeroed by the FMA hardware before the
Welford lift step observes them. The variance appears smaller than the true
variance. Silent wrong answer.

### Why the cache key doesn't catch it

`DenormalMode` lives in `DoorCapability`. The `capability_fingerprint` in
`CacheKey` includes the full `DoorCapability`. So two compile runs where the
DenormalMode changes DO produce different cache keys. That part is correct.

The gap: the `CompiledArtifact` does not carry a record of which `DenormalMode`
it was compiled under. When retrieved from cache, there's no way to verify that
the current capability's DenormalMode matches the artifact's compilation mode.

If the DenormalMode changes between the compile and a later `dispatch()` call
(e.g., another thread modified the MXCSR register on CPU, or a driver update
changed GPU defaults), the cached artifact may be dispatched under a different
DenormalMode than it was compiled under. Silent correctness gap.

### Fix: `DenormalMode` on `CompiledArtifact`

```rust
struct CompiledArtifact {
    binary: Vec<u8>,
    entry_points: Vec<EntryPoint>,
    pipeline_metadata: PipelineMetadata,
    scratch_required: ScratchSpec,
    determinism: DeterminismClass,
    denormal_mode: DenormalMode,  // the mode under which this was compiled
}
```

`dispatch()` checks `artifact.denormal_mode == capability.denormal_mode` before
launching. If they disagree, returns `Err(LaunchError::CapabilityMismatch)`.

## BONUS ATTACK 5 — Shape fingerprint collision: metric-identity gap

**Verdict: Gap exists. Two distinct recipes produce identical Shapes.**

### The break

Recipe A: Euclidean distance matrix over raw features.
Recipe B: Mahalanobis distance matrix over the same features.

Both: `ByKey { keys }` grouping, same dtype, same N, `Validity::Propagate`,
no assumption tags.

Same Shape → same `CacheKey` → same compiled kernel retrieved from cache.

But the kernels ARE different:
- Euclidean: `sqrt((x - y)²)` for each pair.
- Mahalanobis: `sqrt((x-y)ᵀ Σ⁻¹ (x-y))` with `Σ⁻¹` in a separate buffer.

The combine functions are different binaries. Serving the Euclidean kernel for
the Mahalanobis consumer produces a silently wrong distance matrix.

### Why TamSession doesn't save this

TamSession's `is_share_compatible_with()` checks AssumptionTags. Neither recipe
adds a tag distinguishing their metric. TamSession says they're compatible and
serves the wrong kernel. No error, no warning.

### Fix: metric-identity as a first-class AssumptionTag

```rust
pub enum AssumptionTag {
    NoNonFinite,
    SortedAscending,
    Centered,
    UnitNorm,
    Metric(MetricKind),        // new: which distance metric
    Custom(String),
}

pub enum MetricKind {
    Euclidean,
    Mahalanobis,
    Manhattan,
    Cosine,
    // ...
}
```

Distance recipes must emit `AssumptionTag::Metric(MetricKind::Euclidean)` or
`Metric(MetricKind::Mahalanobis)` on their output Shape. The sharing check then
correctly blocks cross-metric sharing. The `Custom(String)` path can carry this
today, but only if recipes are consistently disciplined — there's no enforcement.
First-class variant provides enforcement.

## BONUS ATTACK 6 — Capability fingerprint staleness during mid-run driver update

**Verdict: Partially survives. One gap under Windows WDDM driver hot-swap.**

### What R10′ gets right

`CacheKey::capability_fingerprint` is derived from `DoorCapability` at compile
time. A driver update changes `driver_version` → new capability fingerprint →
cache miss for new compilations → recompile under the new driver. This is
correct for NEWLY compiled kernels.

### The gap: in-flight dispatches under old driver

1. Pipeline step 35 compiles kernel K under driver v1. Cache entry uses v1
   capability fingerprint.
2. Driver updates to v2 mid-run (Windows WDDM permits driver hot-swap for some
   GPU families without device reset on all contexts).
3. Step 36 calls `capability()` → returns v2. Cache miss → recompiles under v2.
4. Step 35's `e1 = dispatch(...)` is still in flight (event not yet waited on).
   The binary was compiled for v1, but the context has been updated to v2.

Behavior depends on whether v2 is backward-compatible with v1 for this binary.
For CUDA PTX: PTX is forward-compatible; older PTX usually runs fine on newer
drivers. But if v2 changes numerical behavior (e.g., fast-math defaults, denormal
handling), step 35 and step 36 may execute under different numerical semantics
within the same pipeline.

### The fix

`capability()` should return a snapshotted capability from pipeline-compile
time, held for the duration of one pipeline execution. TAM (Sweep 23) owns
this snapshot. If the live capability diverges from the snapshot, TAM drains
all in-flight events and restarts with the new capability. R10′ should document
this as a TAM-level concern — the trait itself cannot enforce it, but the design
must acknowledge it.

## Summary

| Attack | Verdict | Severity | Fix |
|--------|---------|----------|-----|
| A1 NPU per-atom lower() | **Breaks** | High | Add `lower_pipeline()` to `DoorCodegen` |
| A2 DeterminismClass | **Breaks** | Medium | Add `SeededDeterministic` 4th variant |
| A3 Validity + async masking | **Breaks** | High | Define stream-poison semantics |
| B4 Validity vs DenormalMode | **Breaks** | Medium | `denormal_mode` field on `CompiledArtifact` |
| B5 Shape metric-identity | **Breaks** | High | First-class `AssumptionTag::Metric` |
| B6 Capability staleness | **Partially survives** | Low | TAM-level snapshot (document, don't redesign) |

### Two fixes that close the most ground

**1.** Add `lower_pipeline()` to `DoorCodegen` with a default impl that calls
`lower()` per-atom. NPU doors override `lower_pipeline()`. Closes A1.

**2.** Define stream-poison semantics in `DoorDispatcher`. After `wait(e)`
returns `Err(Validity)`, the stream is poisoned; `dispatch()` on a poisoned
stream returns `Err(StreamPoisoned)`; `stream_reset()` recovers. Closes A3.

B5 is the highest-severity gap that can be closed without touching the
trait signature — it's a Shape bug, not a DoorBackend bug. First-class
`AssumptionTag::Metric` closes it. Tracked as GAP-B5 for pathmaker.
