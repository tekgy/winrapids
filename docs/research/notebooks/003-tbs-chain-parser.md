# Notebook 003 — .tbs Chain Parser

*2026-03-31 | Navigator | in progress*

---

## Hypothesis

A minimal hand-written recursive descent parser can parse the `.tbs` chain syntax into a typed AST in ~150 lines of Rust, with no new dependencies. The resulting `TbsChain` AST:

1. Round-trips cleanly (parse → display → parse produces identical AST)
2. Is co-native — both humans and AI agents can generate valid `.tbs` text given only the vocabulary reference
3. Maps directly to `TamPipeline` method calls (AST node ↔ method, args ↔ parameters)

The secondary hypothesis: the grammar is simple enough that a correctly structured `.tbs` program can be generated by an LLM given only the vocabulary sketch, without seeing the parser internals. This is the co-native test.

---

## Grammar

```
chain    = step ('.' step)*
step     = name '(' arglist? ')'
name     = IDENT ('.' IDENT)*    // handles "train.linear"
arglist  = arg (',' arg)*
arg      = (IDENT '=')? value
value    = '"' [^"]* '"'         // string
         | '-'? [0-9]+ ('.' [0-9]+)?  // number
         | 'true' | 'false'      // bool
IDENT    = [a-zA-Z_][a-zA-Z0-9_]*
```

Note: no semicolons, no let bindings, no keywords other than `true`/`false`. The minimal subset that covers the vocabulary sketch.

## Design

### AST types

```rust
pub struct TbsChain { pub steps: Vec<TbsStep> }

pub struct TbsStep {
    pub name: TbsName,
    pub args: Vec<TbsArg>,
}

pub enum TbsName {
    Simple(String),              // "normalize"
    Dotted(String, String),      // "train", "linear"
}

pub enum TbsArg {
    Positional(TbsValue),
    Named { key: String, value: TbsValue },
}

pub enum TbsValue {
    Str(String),
    Float(f64),
    Int(i64),
    Bool(bool),
}
```

### Parser approach
Hand-written recursive descent. No `nom`, no `pest`. The grammar has no ambiguity and no left recursion — a direct descent parser is ~10 lines per production.

State: `struct Parser { input: &str, pos: usize }` with `peek()`, `advance()`, `skip_ws()`, `expect_char()`, `parse_ident()`, `parse_value()`, `parse_arg()`, `parse_step()`, `parse_chain()`.

Error handling: `Result<T, TbsParseError>` where `TbsParseError` carries the character position and a message.

### Why no `nom` or `pest`?
- No new dependencies in `Cargo.toml` — tambear is a GPU runtime, not a parsing library
- The grammar is small enough that a hand-written parser is easier to read than combinator chains
- Error messages from a hand-written parser are more useful for a scripting language

### Display/round-trip
`impl Display for TbsChain` emits canonical `.tbs` text. Round-trip test: `parse(display(parse(src))) == parse(src)`.

## Results

**13 new parser tests pass. 138 total (121 unit + 17 doctest), all green.**

| Test | Verifies |
|------|---------|
| `parse_single_step_no_args` | `normalize()` → Simple name, empty args |
| `parse_named_args` | `discover_clusters(epsilon=0.5, min_samples=2)` |
| `parse_dotted_name` | `train.linear(target="price")` |
| `parse_multi_step_chain` | 3-step chain with `.` separators |
| `parse_positional_args` | Positional args indexable as named |
| `parse_bool_value` | `stratified=true` |
| `parse_negative_number` | `clip(min=-1.0, max=1.0)` |
| `parse_comment` | `// line comment` |
| `round_trip` | `parse(display(chain)) == parse(src)` |
| `parse_full_vocabulary_sample` | 7 vocabulary entries from the garden sketch |
| `error_on_missing_paren` | `normalize` (no parens) → Err |
| `error_on_unterminated_string` | Missing closing `"` → Err |
| `error_on_trailing_junk` | Extra text after chain → Err |

**Actual line count**: 310 lines total (AST + display + parser + tests). Parser logic alone: ~140 lines. The "50 lines" estimate was optimistic — the lookahead for dotted names adds ~30 lines of careful state management.

**Surprise**: The dotted name disambiguation is the hardest part. `train.linear(` must parse as `Dotted("train", "linear")`, but `normalize().\ndiscover_clusters(` must parse `normalize` as the name (not `normalize.discover`). The solution: after seeing `IDENT '.' IDENT`, peek at what follows. If it's `(`, it's a dotted name. Otherwise, reset and treat the `.` as a step separator.

The lookahead saves/restores `self.pos` twice in the worst case — still O(1) since idents are bounded.

**Surprise**: `get_arg(key, positional_index)` bridges the named/positional gap cleanly. `step.get_arg("epsilon", 0)` returns the value whether the user wrote `epsilon=0.5` or just `0.5`. This is important for the co-native property — AI agents can use named arguments, humans can use positional, and the executor handles both.

## Discussion

The hypothesis was confirmed: ~150 lines, no dependencies, round-trips correctly.

**Co-native test** (informal): The vocabulary sketch in the garden contains enough information for an LLM to generate valid `.tbs` programs. The chain `normalize().discover_clusters(epsilon=0.5, min_samples=2).train.linear(target="price")` was written in the sketch before this parser existed, and it parses correctly without modification.

**What the parser is missing:**
- No `read("path")` source step execution (the chain starts from an implicit data source)
- No interpreter/executor (AST → `TamPipeline` calls)
- No source location in error messages (position but no line/column)

The interpreter is the next piece. Given:
```rust
pub fn execute(chain: &TbsChain, data: Vec<f64>, n: usize, d: usize, y: Option<&[f64]>)
    -> Result<(TamPipeline, Option<LinearModel>), Box<dyn std::error::Error>>
```

Each step maps directly to a `TamPipeline` method call. The step name dispatch is a `match` on `step.name.to_string()`. Arguments are extracted via `get_arg(key, index)`. The interpreter closes the loop: `.tbs` text → parsed chain → executed pipeline → results.

**The grammar is stable.** Every vocabulary entry from the garden sketch parses correctly. Adding new operations requires only adding them to the vocabulary — the parser handles any `IDENT(args...)` without modification.
