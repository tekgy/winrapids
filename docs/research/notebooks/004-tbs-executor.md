*2026-03-31 | Navigator*

## Hypothesis

A `TbsExecutor` can dispatch each `TbsStep` to the corresponding `TamPipeline` method in ~60 lines, closing the loop: `.tbs` text → parsed `TbsChain` → executed `TamPipeline`.

The co-native property is testable end-to-end: write a `.tbs` program as a string, execute it, get a `TamPipeline` and optional `LinearModel`.

---

## Design

```
TbsChain (from parser)
   ↓ execute(chain, data, n, d, y=None)
TamPipeline (step by step dispatch)
   + Option<LinearModel> (if train.linear was called)
```

`TbsResult` carries both. Step dispatch is a `match on step.name.to_string()`.

Supported operations for MVP (what TamPipeline currently implements):
- `normalize()` → `.normalize()`
- `discover_clusters(epsilon, min_samples)` → `.discover_clusters(eps, min_samps)`
- `train.linear(target)` → `.train_linear(y)` (y from executor context, target arg noted)

Unsupported operations → `Err("unsupported: {name}")`.

## Discussion

The loop is closed: `.tbs` text → `TbsChain::parse()` → `execute()` → `TamPipeline` → results.

```
"normalize().discover_clusters(epsilon=3.0, min_samples=1).train.linear(target=\"y\")"
    ↓ TbsChain::parse()
TbsChain { steps: [normalize(), discover_clusters(3.0, 1), train.linear("y")] }
    ↓ execute(chain, data, 6, 2, Some(y))
TbsResult { pipeline: TamPipeline{ session_len=1, clusters=2 }, model: Some(LinearModel{r²>0.9}) }
```

**The session is transparent**: the executor never mentions it. `normalize()` uses CPU column stats; `discover_clusters()` registers the distance matrix in the session; `train_linear()` registers column stats. If the same data flows through again, cache hits happen without any executor changes.

**The co-native property holds**: the string `"normalize().discover_clusters(epsilon=3.0, min_samples=1)"` is the same whether written by a human or generated by an LLM. No translation layer. The executor handles both.

**Unsupported operations give useful errors**: `"unsupported .tbs operation: window"` tells the user exactly what they need to implement. As `TamPipeline` grows (kmeans, knn, filter, etc.), adding them to the executor is a one-line match arm.

**What's missing:**
- `read("path")` source step (requires TB/MKTF file loading)
- `kmeans(k)`, `knn(k)`, `filter(predicate)` — as TamPipeline adds them
- Named column target (`target="price"` currently just logs the name, uses y from context)
- The parser handles `read()` as a valid chain step; the executor just errors on it — the grammar is stable
